# 密码变量契约与 Initial Credentials File 设计

状态：`READY_FOR_IMPLEMENTATION`

## 目标

统一 LightAI Go 中 server、启动脚本、E2E、bootstrap 工具、安装文档对管理员密码的理解，避免以下问题：

- 初始密码和最终密码混用；
- 旧环境变量与新环境变量冲突；
- reset DB 后无法追溯实际初始化密码；
- 使用环境变量初始化后，进程重启无法从 runtime 目录找回初始凭据；
- 日志或状态文件泄露明文密码、token、cookie、CSRF。

## 术语

| 术语 | 含义 |
|---|---|
| Initial password | clean DB / 首次启动后，admin 当前可登录的初始密码 |
| Final/admin password | bootstrap 首次登录后要修改成的最终管理员密码 |
| Initial credentials file | server 在 runtime 目录下写出的初始化凭据留痕文件 |
| Clean DB initialization | DB 无 admin 或处于首次 bootstrap admin 创建状态 |
| Existing DB | DB 已有 admin，不应再按当前 env 覆盖密码或 credentials file |

## 正式环境变量契约

### `LIGHTAI_BOOTSTRAP_INITIAL_PASSWORD`

含义：初始管理员密码。**Canonical initial password 变量。**

用途：

- server clean DB 首次初始化 admin 用户时读取（优先级最高，仅次于内部 cfg.Password）；
- bootstrap 在最终密码登录失败后，用它尝试初始登录；
- reset DB / 首次启动场景使用；
- 安装部署时可用它预置初始 admin 密码。

### `LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD`

含义：目标管理员密码 / 最终管理员密码。**Canonical final/admin password 变量。**

用途：

- bootstrap 首次登录后，如果系统要求改密，将 admin 密码修改为该值；
- 后续重复运行 bootstrap、Browser E2E、API E2E 优先使用该值登录；
- 历史代码中如果曾把它作为初始密码使用，只能作为 legacy fallback，并输出 WARN。

该变量**不是** canonical initial password 变量。

## A. Server-Side Password Resolution

适用范围：`internal/server/auth/bootstrap.go` `InitBootstrap()` — server clean DB 首次初始化 admin 用户时的密码解析。

以下顺序以**当前 Go 代码实际逻辑为准**（实现时间：2026-06-25）：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | `cfg.Password` | `BootstrapConfig.Password` 字段，通常为 `""`（硬编码空字符串），这是 Go 内部 config 层字段，不是用户主要操作入口 |
| 2 | `LIGHTAI_BOOTSTRAP_INITIAL_PASSWORD` | Canonical initial password 环境变量 |
| 3 | `LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD` | Legacy fallback，输出 `log.Warn("bootstrap using deprecated password env", ...)` |
| 4 | existing `runtime/initial-credentials.txt` | 复用已有凭据文件中的密码，不重新生成 |
| 5 | auto-generate | `crypto/rand` 生成 32 字符 hex，设置 `autoGenerated=true`，写入 credentials file |

**Server 侧不支持 CLI flag、profile、password file。** 这些属于 bootstrap script 侧功能，通过环境变量透传给 server。

**无论密码来自哪个优先级**（env、legacy fallback、credentials file、auto-generate），只要本次创建了 admin 用户（`userCreated=true`），**必须调用 `writeInitialCredentials()` 写入本次实际生效的密码**。写入规则见 §[Credentials File 规则]。

**Server restart 不覆盖 existing credentials file。** 如果 DB/admin 已存在（`INSERT OR IGNORE` 无新行），不执行写入。

## B. Bootstrap-Script-Side Password Resolution

适用范围：`scripts/lightai-bootstrap.sh` — bootstrap 脚本在登录、首次改密、后续登录时使用。

脚本侧支持 CLI flag、环境变量、profile、password file 等多种来源，通过登录 API / 改密 API 使用这些密码。

### Initial Password（bootstrap 首次登录使用）

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | `--initial-password` | 命令行直接传入初始密码 |
| 2 | `LIGHTAI_BOOTSTRAP_INITIAL_PASSWORD` | Canonical initial password 环境变量 |
| 3 | `profile.auth.initial_password_env` 指向的环境变量 | profile 中指定的间接环境变量名 |
| 4 | `--initial-password-file` | 命令行指定的密码文件 |
| 5 | `profile.auth.initial_password_file` | profile 中指定的密码文件路径 |
| 6 | runtime-dir 下 existing initial credentials file | `readPasswordFromCredentialsFile()` 逻辑复用 |
| 7 | `profile.auth.initial_password` | profile 中的低优先级硬编码值（不推荐生产使用） |
| 8 | server default / auto-generated | 如果没有以上任何来源，依赖 server auto-generate 写入的 credentials file |

### Final/Admin Password（bootstrap 改密和后续登录使用）

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | `--admin-password` | 命令行直接传入目标密码 |
| 2 | `LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD` | Canonical final/admin password 环境变量 |
| 3 | `profile.auth.final_password_env` 指向的环境变量 | profile 中指定的间接环境变量名 |
| 4 | `--admin-password-file` | 命令行指定的密码文件 |
| 5 | `profile.auth.final_password_file` | profile 中指定的密码文件路径 |

**CLI 优先级最高，profile 优先级最低。** Bootstrap script 先按 CLI → env → profile → default 合并参数，再按上述优先级解析密码。

## Legacy Fallback 策略

`LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD` 在 server 侧作为初始密码的 legacy fallback：

1. fallback 行为已实现（server `InitBootstrap()` Priority 3）。
2. fallback 输出 `log.Warn`。
3. fallback 不覆盖 `LIGHTAI_BOOTSTRAP_INITIAL_PASSWORD`（Priority 2 先于 Priority 3）。
4. 新文档、新 profile、新脚本使用 `LIGHTAI_BOOTSTRAP_INITIAL_PASSWORD` 表示初始密码。
5. `LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD` 在未来版本可能移除初始密码 fallback 行为（届时只保留为目标密码变量）。

## Initial Credentials File 规则

### 写入原则

clean DB 首次初始化 admin 用户时（`userCreated=true`），**无论初始密码来自哪个优先级**，都必须把"本次实际生效的初始密码"写入 runtime initial credentials file。

明确覆盖以下场景：

- `LIGHTAI_BOOTSTRAP_INITIAL_PASSWORD` 提供了密码 → **写入** credentials file
- legacy `LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD` fallback → **写入** credentials file（同时已输出 WARN）
- auto-generate 生成随机密码 → **写入** credentials file
- existing credentials file 被复用 → **不重复写入**（文件已存在，`writeInitialCredentials` 检测到已存在则跳过）

### 复用原则

如果没有 env（Priority 2、3 都无值），且 existing credentials file 存在，则**复用** existing credentials file 中的密码（`readPasswordFromCredentialsFile()`），不重新生成。

### 不覆盖原则

1. DB/admin 已存在时（`INSERT OR IGNORE` 返回 0 rows affected），不因当前 env 存在而覆盖 credentials file。
2. credentials file 已存在且本次没有重新初始化 DB，不覆盖。
3. server restart 不覆盖 existing credentials file。
4. bootstrap 脚本不直接改写 credentials file，除非明确执行 server 侧初始化或有专门安全命令。

### Reset DB 规则

reset DB 后重新初始化 admin，应写入本次实际生效初始密码。

如果 env password 与 existing credentials file 冲突，但本次是 clean DB 初始化且 env 优先级更高：

1. 备份旧 credentials file 为 `<file>.bak.<timestamp>`。
2. 写入本次实际生效密码。
3. 记录 WARN（不打印明文密码）。

### 写入失败处理

如果 admin 已创建但 credentials file 写入失败：

- server 输出明确 ERROR 或 WARN（当前实现：`log.Warn("failed to write initial credentials file", ...)`）。
- 日志说明 credentials file missing/incomplete。
- **不打印明文密码。**
- 测试覆盖该风险路径。

## Credentials File 路径与格式

当前代码实际路径（`bootstrap.go:280`）：

```go
credPath := "runtime/initial-credentials.txt"
```

这是相对于 server 工作目录的相对路径。实际运行时位于：

```text
<server-working-dir>/runtime/initial-credentials.txt
```

文件格式（当前代码实际格式）：

```text
============================================
LightAI Go - Initial Credentials
Generated: <RFC3339 timestamp>
============================================

[Web/Admin]
Username: admin
Password: <hex-string>

[Grafana]
See: runtime/initial-credentials.txt (appended by start-observability.sh)
Note: If Grafana is in use, start observability to record its password.
```

该文件由 `writeInitialCredentials()` 生成。文件已存在时不覆盖（通过 `os.OpenFile` 的 `O_EXCL` flag）。

## 文件安全

1. credentials file 权限为 `0600`（`os.OpenFile` 指定 `0600`）。
2. 父目录不存在时自动创建（`os.MkdirAll(dir, 0700)`）。
3. 文件包含敏感信息，纳入 `.gitignore` 或不提交到 git。
4. 日志不打印明文密码。
5. `auth.json`、`effective-config.json`、`bootstrap-state.json` 不输出明文密码、token、cookie、CSRF。
6. 安装交付时不要把 credentials file 打包或上传到不受控位置。

## Server / Bootstrap / E2E 行为关系

### Server clean DB 初始化

1. 按 §A Server-Side 优先级解析 initial password。
2. 创建 admin 用户（`INSERT OR IGNORE`）。
3. 如果 `userCreated=true`：调用 `writeInitialCredentials()` 写入 credentials file。
4. 如果 `ForceChangePassword=true`：用户 `must_change_password=1`。

### Bootstrap 首次运行

1. 使用 final/admin password（§B Final/Admin Password 优先级）尝试登录。
2. 登录成功：说明已经初始化或改密完成 → 直接输出 auth.json。
3. 登录失败：使用 initial password（§B Initial Password 优先级）读取初始密码。
4. 使用 initial password 登录。
5. 如果 `must_change_password=true`，使用 final/admin password 调用 `POST /api/v1/auth/change-password`。
6. 改密后用 final/admin password 重新登录。
7. 如果 `must_change_password=true` 但 final/admin password 缺失：输出 `FAIL_MISSING_FINAL_PASSWORD`。

### Browser/API E2E

后续 E2E 使用 `LIGHTAI_BOOTSTRAP_ADMIN_PASSWORD` 登录，不再使用 initial password。
