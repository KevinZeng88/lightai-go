# Runtime Operations UX & Resource Controls — Implementation Plan

> Date: 2026-06-23  
> Status: Proposed implementation sequence. Claude/Codex must review before modifying code.

## 1. Required review-first workflow

This is a design-to-implementation plan. It must not be executed blindly.

Required workflow:

1. Read:
   - `docs/design/runtime-operations-ux-resource-controls.md`
   - `docs/reports/phase-3/runtime-operations-ux-resource-controls/00-known-issues-and-evidence.md`
   - this implementation plan
   - `docs/reports/phase-3/runtime-operations-ux-resource-controls/02-verification-and-acceptance-plan.md`
2. Inspect current code and identify mismatches with file paths, APIs, and existing abstractions.
3. Write a review report before any code modification.
4. Ask for approval if schema change, route naming change, or major frontend layout rewrite is required.
5. Implement in phases.
6. Run phase-specific tests after each phase.
7. Write closeout and commit/push only after all acceptance criteria are met.

## 3. Phase 1 — RunPlan lint

### Goal

Detect command/env conflicts before container start.

### Candidate files

Create or update:

```text
internal/server/runplan/lint.go
internal/server/runplan/lint_test.go
```

If the repository already has an equivalent location, use the existing package and document the choice.

### Required capabilities

- Parse CLI args into flags and values.
- Parse env vars.
- Detect duplicate CLI flags.
- Detect env/CLI conflicts for the same logical parameter.
- Detect platform-owned arguments overridden by user extra args.
- Detect high-risk Docker/container settings.
- Return `RunPlanLintResult`.

### Required initial logical parameter specs

llama.cpp:

- `host`: `--host`, `LLAMA_ARG_HOST`, platform-owned, reject conflict
- `port`: `--port`, `LLAMA_ARG_PORT`, platform-owned, reject conflict
- `model_path`: `-m`, `--model`, platform-owned if generated by system
- `ctx_size`: `--ctx-size`, user-owned
- `gpu_layers`: `--n-gpu-layers`, `--gpu-layers`, user-owned

vLLM:

- `host`: `--host`, platform-owned
- `port`: `--port`, platform-owned
- `model_path`: `--model`, platform-owned
- `served_model_name`: `--served-model-name`, user or platform-owned depending on current design
- `gpu_memory_fraction`: `--gpu-memory-utilization`, user-owned but single-source only

SGLang:

- `host`: `--host`, platform-owned
- `port`: `--port`, platform-owned
- `model_path`: `--model-path`, platform-owned
- `served_model_name`: `--served-model-name`, user or platform-owned depending on current design
- `gpu_memory_fraction`: `--mem-fraction-static`, user-owned but single-source only

### Required tests

Add tests for:

- llama.cpp `LLAMA_ARG_HOST` + `--host` → error
- llama.cpp `LLAMA_ARG_PORT` + `--port` → error
- duplicated `--ctx-size` → error
- duplicated vLLM `--gpu-memory-utilization` → error
- duplicated SGLang `--mem-fraction-static` → error
- privileged / ipc host / unsafe security-opt → warning
- clean vLLM RunPlan → ok
- clean SGLang RunPlan → ok
- clean llama.cpp RunPlan → ok

### API integration

RunPlan preview / preflight response should include:

```json
{
  "lint": {
    "status": "ok|warning|error",
    "findings": []
  }
}
```

If errors exist, default behavior should block start unless the existing product already has an explicit risk override mechanism.

## 4. Phase 2 — resource controls and shared-GPU admission

### Goal

Expose backend-specific resource controls and detect unsafe shared-GPU placements.

### Candidate files

Create or update:

```text
internal/server/runplan/resource_controls.go
internal/server/runplan/resource_admission.go
internal/server/runplan/resource_admission_test.go
```

### Resource controls

Start with catalog/seed JSON. Do not add schema unless review proves JSON is insufficient.

vLLM:

- `gpu_memory_fraction` → `--gpu-memory-utilization`
- `max_model_len` → `--max-model-len`
- `max_num_seqs` → `--max-num-seqs`
- `max_num_batched_tokens` → `--max-num-batched-tokens`

SGLang:

- `gpu_memory_fraction` → `--mem-fraction-static`
- `max_running_requests` → `--max-running-requests`
- `chunked_prefill_size` → `--chunked-prefill-size`
- `kv_cache_dtype` where supported
- `attention_backend` → `--attention-backend`

llama.cpp:

- no `gpu_memory_fraction`
- `gpu_layers` → `--n-gpu-layers` / `--gpu-layers`
- `ctx_size` → `--ctx-size`
- `batch_size` → `--batch-size`
- `ubatch_size` → `--ubatch-size`
- `cache_type_k` → `--cache-type-k`
- `cache_type_v` → `--cache-type-v`
- `split_mode` → `--split-mode`
- `main_gpu` → `--main-gpu`
- `tensor_split` → `--tensor-split`

Ollama:

- no platform memory budget control in this batch

### Admission rules

- exclusive existing instance blocks new placement on same GPU;
- requested exclusive placement blocks if any active instance uses the GPU;
- vLLM/SGLang memory fraction participates in declared budget sum;
- sum > 1.0 blocks by default;
- unknown budget creates warning in shared mode;
- explicit oversubscribe override allowed only if existing audit mechanism supports it; otherwise defer override and block.

### Required tests

- two vLLM instances on same GPU: 0.5 + 0.5 → ok
- two vLLM instances on same GPU: 0.7 + 0.7 → blocked
- vLLM 0.4 + SGLang 0.5 → ok
- llama.cpp shared GPU → warning unknown budget
- exclusive existing instance + new request → blocked
- requested exclusive + existing active instance → blocked
- oversubscribe override, if implemented → warning + audit requirement

## 5. Phase 3 — runtime log classifier

### Goal

Classify known warnings and errors so users are not forced to read raw logs.

### Candidate files

Create or update:

```text
internal/server/runplan/log_classifier.go
internal/server/runplan/log_classifier_test.go
internal/server/runplan/testdata/runtime-logs/*.log
```

If runtime log handling already belongs to another package, use that package and document the choice.

### Required fixtures

- `sglang-torchao-syntax-warning.log`
- `sglang-attention-backend-default.log`
- `llamacpp-env-host-overwritten.log`
- `cuda-oom.log`

### Required classifications

- SGLang torchao syntax warning → `noise` or `advisory`
- SGLang attention backend default → `advisory`
- llama.cpp host env overwritten → `warning`
- CUDA OOM → `error`
- fatal startup tracebacks → `error` or `fatal`

### API integration

Model test results and/or instance diagnostics should return:

```json
{
  "classified_log_events": []
}
```

Do not let `noise` or `advisory` change instance lifecycle state.

## 6. Phase 4 — model instance auto-refresh

### Goal

Model instance status should update without manual page refresh.

### Candidate files

Create or update:

```text
web/src/composables/useAutoRefresh.js
web/src/composables/useInstanceStatusPolling.js
```

Use `.ts` if the web project uses TypeScript.

### Behavior

- transitional states: 2–3 seconds
- stable states: 15–30 seconds
- document hidden: pause or 60 seconds
- route leave: stop polling
- request failures: backoff
- visible UI: last refreshed timestamp and manual refresh button

### Optional API

If current list endpoint is heavy, add:

```text
GET /api/v1/model-instances/status-summary
```

Only add this if the review shows existing endpoints are too heavy or noisy.

### Required tests

- transitional state triggers frequent polling;
- stable state slows polling;
- document hidden pauses or slows polling;
- polling stops on unmount;
- request failure shows stale warning and backs off.

## 10. Approval gates

Implementation must pause for user approval if any of the following is discovered:

- DB schema change is required;
- WebSocket is required instead of polling;
- full arg abstraction layer is required;
- resource controls cannot fit existing BackendVersion/catalog JSON;
- status-summary API would require large backend refactor;
- configuration layout rewrite touches more than two major pages;
- real Docker/GPU E2E becomes necessary for default verification;
- any old config path is still referenced and deletion is risky.

## 11. Commit policy

- Do not create a new branch unless explicitly instructed.
- Commit only after final closeout is correct.
- Push to current branch.
- Final status must be clean.
