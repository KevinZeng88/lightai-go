## 1. Issue

MetaX runtime template/device binding needs to be clearly separated from NVIDIA GPU IDs model. NVIDIA uses `gpu_ids` → `CUDA_VISIBLE_DEVICES` → Docker GPU DeviceRequest. MetaX uses raw device paths + privileges OR metax-docker `--gpus` selector.

## 3. Binding Rules Per Vendor

### NVIDIA
- Uses `gpu_ids` from placement → `CUDA_VISIBLE_DEVICES` → Docker GPU DeviceRequest
- Requires GPU assignment; preflight blocks if no GPU resolved
- Docker preview: `--gpus "device=0"`, `-e CUDA_VISIBLE_DEVICES=0`

### MetaX (native Docker device-path mode)
- Uses raw device paths: `/dev/dri`, `/dev/mxcd`, `/dev/infiniband`
- Requires: `privileged=true`, `group_add: [video]`, `ipc_mode: host`
- Env: `MACA_VISIBLE_DEVICE` (not CUDA)
- Security: `seccomp=unconfined`, `apparmor=unconfined`
- Docker preview: `--device /dev/dri:/dev/dri --device /dev/mxcd:/dev/mxcd --privileged --group-add video --ipc host --security-opt seccomp=unconfined -e MACA_VISIBLE_DEVICE=N`

### MetaX (metax-docker selector mode)
- Template-supported, driver-pending (requires MetaX Docker runtime)
- Uses `--gpus=all` or `--gpus=[ID,SN]` selector via metax-docker
- No NVIDIA-style DeviceRequest
- Documented for future implementation

### CPU
- No GPU assignment required
- No GPU env vars
- No device passthrough
