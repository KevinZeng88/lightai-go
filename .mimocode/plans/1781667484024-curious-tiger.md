# TOTP-Based MFA Implementation Plan

**Goal:** Add optional TOTP-based multi-factor authentication to LightAI Go with two-step login, backup codes, and admin management.

## Architecture

Two-step login flow: `POST /login` returns `428 Precondition Required` with `mfa_required` when TOTP is enabled, client then calls `POST /login/mfa-verify` to complete login. TOTP secrets stored encrypted in SQLite. 10 one-time backup codes per user (bcrypt-hashed). Server generates SVG QR codes. Admin can enable/disable MFA per user.

## Tech Dependencies

- `github.com/pquerna/otp` ŌĆö TOTP generation/verification
- `github.com/skip2/go-qrcode` ŌĆö QR code SVG generation
- `golang.org/x/crypto/bcrypt` ŌĆö backup code hashing (already in go.sum via argon2)
- Go stdlib `crypto/aes` + `crypto/cipher` ŌĆö AES-256-GCM encryption for TOTP secrets

## Chunk 4: MFA Handlers

**New file:** `internal/server/auth/mfa_handlers.go`

Struct: `MFAHandler { DB *db.DB, EncKey []byte }`

Endpoints:
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/v1/mfa/status` | Session | Returns `mfa_enabled`, `enrolled` |
| POST | `/api/v1/mfa/enroll` | Session | Generates TOTP key, QR, backup codes |
| POST | `/api/v1/mfa/verify-setup` | Session | Verifies TOTP code, enables MFA |
| POST | `/api/v1/mfa/disable` | Session | Disables MFA (requires password + TOTP code) |

Key flows:
- **Enroll**: Generate TOTP key ŌåÆ encrypt secret ŌåÆ store in `user_mfa` ŌåÆ generate backup codes ŌåÆ return QR + codes
- **Verify setup**: Get encrypted secret ŌåÆ decrypt ŌåÆ verify code ŌåÆ set `mfa_enabled = 1`
- **Disable**: Verify password ŌåÆ if MFA enabled, verify TOTP code ŌåÆ delete MFA data

**Test:** `internal/server/auth/mfa_handlers_test.go` ŌĆö status checks with/without MFA

## Security Considerations

1. **TOTP secret encryption**: AES-256-GCM with random nonces. Key persisted to `mfa_encryption.key` (mode 0600).
2. **Backup codes**: bcrypt-hashed, one-time use, 10 per enrollment.
3. **Rate limiting**: Applied to both `/login` and `/login/mfa-verify`.
4. **No session created until MFA verified**: 428 response does not create a session.
5. **Admin break-glass**: Platform admin can force-disable MFA for any user.
6. **Clock drift**: `pquerna/otp` uses ┬▒1 window (30s per step) by default.

## Migration Safety

- V11 uses `ALTER TABLE ... ADD COLUMN` (SQLite ignores if column exists)
- `CREATE TABLE IF NOT EXISTS` is idempotent
- Existing users get `mfa_enabled = 0` ŌĆö no behavior change
